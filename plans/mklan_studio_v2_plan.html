1 · Async Job Queue with Real-time Progress

Problem now: Training runs, gallery scans, caption jobs, and image generation all use a custom SQLite-backed job manager that the frontend polls. This creates noisy HTTP traffic, stale UI states, and makes cross-module job visibility hard to build.

v2 approach: Introduce ARQ (async Redis Queue) or Celery + Redis as the task broker. Every long-running task (LoRA training, ComfyUI job, media scan, caption scan) becomes a proper async worker task. The frontend subscribes to a Server-Sent Events (SSE) stream or a lightweight WebSocket channel for live progress, log tail, and completion events — no more polling.

Key benefits: Retries, priority queues, dead-letter handling, cross-module job dashboard in one shared view. Redis already lives next to Postgres in Docker Compose — adding it is one service entry.

2 · PostgreSQL Migration for Core Backend Data

Problem now: Studio's main data (wildcards, movie scripts, cards, training runs) lives in separate SQLite files. SQLite has no concurrent writer support, no vector search, and no FTS beyond FTS5. The Media Indexer already runs Postgres + pgvector — but that's isolated to the Gallery module.

v2 approach: Move all core FastAPI models to Postgres using SQLModel + Alembic migrations. Consolidate into a single schema with per-module namespacing (prefixed tables or schemas). This unlocks: full-text search across wildcards, movie beats, and card lore in one query; pgvector similarity search for "find images similar to this prompt"; and safe concurrent writes from multiple workers.

Migration strategy: Write a one-time migration script using the existing backend/migrations/ tooling. Keep SQLite as fallback behind a feature flag for the first release.

3 · Full ComfyUI Workflow Engine

Problem now: The Generation module dispatches basic ComfyUI-compatible image jobs with a fixed node graph. There's already a plans/v2-comfyui-image-generator.md in the repo — this is the planned direction. Users who want ControlNet poses, IP-Adapter style transfer, or Hires Fix must leave Studio entirely and go to the ComfyUI UI.

v2 approach: Build a Workflow Template Library inside Studio. Store ComfyUI API-format JSON templates in data/integrations/comfyui/. The UI lets users pick a template (txt2img, img2img, ControlNet openpose, IP-Adapter, upscale), fill in the variable fields (prompt, strength, seed, checkpoint, LoRA weights), and dispatch. The backend validates against a schema and sends the assembled workflow to the ComfyUI API endpoint. Add a result websocket listener that streams ComfyUI's native progress events into the shared job queue from upgrade #1.

Phase 2: A simple visual node inspector (read-only) showing which nodes are in the active workflow — not a full editor, just transparency.

4 · Flux + SDXL Pony Training Support

Problem now: The Training module is built around SDXL LoRA via Kohya-ss. Flux has become the dominant local image model architecture in 2025 — Flux.1 Dev/Schnell and its LoRA variants are now what most serious creators train. The existing training UI assumes SDXL dataset structure and captioning conventions.

v2 approach: Add a model family selector at the top of the Training module (SDXL / SDXL Pony / Flux.1 / SD 1.5). Each family configures: default trainer backend (Dockerfile.gpu for SDXL Kohya, SimpleTuner or Kohya Flux branch for Flux), dataset caption format (natural language for Flux vs tag-based for SDXL), recommended resolution and batch sizes, and trigger word conventions. The caption scan providers already in the UI need a Flux-specific mode (longer natural-language captions, no Danbooru tag vocabulary).

Data directory addition: data/models/images/Flux/ alongside Base/ for SDXL checkpoints.

5 · Video Generation Module

Opportunity: The data layout already has data/models/video/ — it's a placeholder waiting for a UI. Local video diffusion (Wan2.1, CogVideoX-5B, AnimateDiff) is now practical on consumer GPUs. The Movie Script module already generates image/video prompts per scene — connecting those to an actual video backend closes a huge loop.

v2 approach: Add a /video route (new nav item). The module supports: text-to-video with a prompt + model selector; image-to-video (upload a keyframe or pick from Gallery); duration and resolution controls; and a clip browser backed by data/generated/video/. ComfyUI can serve as the video backend via the workflow engine from upgrade #3 (using AnimateDiff or Wan nodes), or a direct VideoCrafter/Wan2.1 API endpoint configurable in Settings.

Movie integration: Each scene card in the Movie module gets a "Generate video clip" button that opens a pre-filled Video job with the scene's prompt, characters, and style tags already populated.

New: /video route Movie module Gallery (clip indexing) data/models/video/ data/generated/video/

6 · Cross-Module AI Copilot Layer

Problem now: The LLM chat in Generation and the Scenario Assistant in Movie are isolated. If you're editing a wildcard, you can't ask the AI to suggest variations. If you're reviewing training captions, there's no in-context AI help. The local LLM endpoint is already configured — it's just not surfaced everywhere.

v2 approach: Build a lightweight Copilot sidebar (collapsible panel, available in every module) powered by the same OpenAI-compatible LLM endpoint. Each module injects its own context into the system prompt: the Wildcards module sends the current wildcard category and 10 existing entries; the Training module sends the caption being reviewed plus the dataset style; the Movie module sends the active scene beat. The sidebar offers context-appropriate suggestions — "3 more wildcard variants", "improve this caption for Flux training", "write a beat between these two scenes".

Auto-caption upgrade: In the Training module, add a Florence-2 / BLIP-2 option that runs locally in a Docker sidecar and produces captions directly in the sidebar — no external API needed.

UX & DevEx Upgrade
7
Multi-Project Workspace Profiles Upgrade
Lightweight named workspaces so different creative projects (a sci-fi movie, a fantasy character pack, a portrait LoRA) each get isolated wildcard libraries, model preferences, movie scripts, and card vaults — without running separate Docker stacks.

7 · Multi-Project Workspace Profiles

Problem now: All of Studio's data lives in flat directories under data/. If you're working on two different creative projects — say a sci-fi movie and a portrait LoRA series — their wildcards, training datasets, and card vaults all mix together. The only isolation today is file naming conventions.

v2 approach: Introduce a Workspace concept: a named profile (stored as a row in Postgres, from upgrade #2) that scopes all module data. Switching workspaces swaps the active wildcard source set, the movie project, the card vault, the preferred checkpoint and LoRA weights, and the ComfyUI workflow templates. The UI shows the active workspace name in the top nav with a quick-switch dropdown. Gallery and generated images remain global (shared pool) but can be filtered by workspace tag.

Implementation: A workspace_id foreign key added to every core table. A minimal Workspace CRUD page under Settings. No authentication required — this is still a single-user trusted-local tool. Import/export a workspace as a ZIP for backup or sharing between machines.
